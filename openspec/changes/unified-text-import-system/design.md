## Context

当前 SailZen 项目的文本导入系统存在以下架构问题：

1. **前端限制**: 现有的 `text_import_dialog.tsx` 组件直接将文件内容读取到内存，通过 HTTP POST 一次性发送到后端，导致大文件（>10MB）上传失败
2. **后端处理瓶颈**: 现有的同步导入 API (`/api/v1/text/import/`) 将整个文本加载到内存解析，没有流式处理或分块机制
3. **AI 工具孤岛**: AI 文本导入工具（`.agents/skills/sailzen-ai-text-import/`）是独立的 Python 脚本，未与主应用集成
4. **缺乏进度反馈**: 长时间导入过程中用户无法获知进度，体验差
5. **任务调度待完善**: 虽然已有 `UnifiedAgentScheduler`，但主要用于大纲提取等分析任务，未扩展支持文本导入

现有架构的优势：
- 数据模型已支持 Work-Edition-DocumentNode 三级结构
- 已有 LLM 客户端和多种 Provider 支持
- 任务调度器框架已建立
- WebSocket 基础设施已存在

## Goals / Non-Goals

**Goals:**
- 支持 GB 级文本文件的可靠上传（分片上传）
- 实现后台异步处理，不阻塞用户操作
- 集成 AI 章节识别，支持非标准格式和特殊章节
- 提供实时进度反馈（上传进度、解析进度、AI 处理进度、存储进度）
- 文本预清理：自动处理编码、移除广告、规范化格式
- 异常检测和报告：识别超长/超短章节、未切分内容
- 复用现有任务调度器架构，保持一致性
- 保持向后兼容：现有同步导入 API 继续工作

**Non-Goals:**
- 不支持非文本文件（如 PDF、EPUB 转换）
- 不实现分布式多机处理（单机异步即可）
- 不添加新的数据库表（复用现有 IngestJob 和 UnifiedAgentTask）
- 不重新设计数据模型
- 不实现章节内容的富文本编辑
- 不处理图片、附件等多媒体内容

## Decisions

### 1. 文件上传方案：分片上传 vs 流式传输

**决策**: 采用流式传输（Streaming Upload）+ 临时文件存储

**理由**:
- 流式传输实现简单，前端使用 Fetch API 的 ReadableStream 即可
- 分片上传需要额外的分片合并逻辑，增加复杂度
- 使用临时文件存储上传内容，处理完成后清理
- 适合文本文件特性（可逐行读取处理）

**替代方案**: 分片上传（Multipart Upload）
- 更适合大型二进制文件
- 支持断点续传
- 但实现复杂，需要维护分片状态

### 2. 任务类型设计：复用 UnifiedAgentTask vs 专用 IngestJob

**决策**: 复用 UnifiedAgentTask，扩展任务类型

**理由**:
- 统一任务管理界面，用户无需学习多套系统
- UnifiedAgentTask 已支持优先级、进度跟踪、状态管理
- IngestJob 表仅用于简单记录，不作为主调度
- 在 UnifiedAgentTask.payload 中存储导入特有配置

**替代方案**: 扩展 IngestJob 表
- 需要重新实现调度逻辑
- 增加系统复杂度
- 但可能更专注

### 3. 章节解析策略：前端解析 vs 后端解析 vs 混合

**决策**: 后端解析，前端仅展示进度

**理由**:
- 后端可利用 LLM 进行智能解析，前端无法直接调用
- 后端可更好地处理大文件和内存管理
- 解析逻辑集中，易于维护和升级

**替代方案**: 前端预解析
- 可以减少服务器负载
- 但无法使用 AI 能力
- 大文件会导致浏览器卡顿

### 4. AI 章节识别集成方式

**决策**: 将 AI 文本导入工具重构为可调用模块

**理由**:
- 保持现有脚本的独立性（仍可独立运行）
- 提取核心逻辑为可导入的 Python 模块
- 后端直接调用模块函数，无需子进程

**集成架构**:
```
sail_server/utils/text_import/
├── __init__.py
├── text_cleaner.py       # 从 skill 迁移
├── ai_chapter_parser.py  # 从 skill 迁移
├── chapter_types.py      # 章节类型定义
└── noise_patterns.py     # 噪音模式
```

### 5. 进度跟踪粒度

**决策**: 四级进度跟踪

**阶段定义**:
1. **上传阶段** (0-20%): 文件从客户端传输到服务器
2. **预处理阶段** (20-40%): 编码检测、文本清理、广告移除
3. **章节解析阶段** (40-80%): AI 分析章节结构、正则切分、特殊章节识别
4. **存储阶段** (80-100%): 创建 Work/Edition、批量插入 DocumentNode

**理由**:
- 提供足够的细节让用户了解处理状态
- 每个阶段有明确的开始和结束
- 便于故障排查和性能分析

### 6. WebSocket 通知策略

**决策**: 使用现有 WebSocket 管理器，扩展消息类型

**消息类型**:
- `import_task_created`: 任务创建成功
- `import_progress_update`: 进度更新（包含阶段和百分比）
- `import_stage_complete`: 某个阶段完成
- `import_completed`: 导入成功完成
- `import_failed`: 导入失败（含错误信息）

**理由**:
- 复用现有基础设施
- 前端已有 WebSocket 连接（用于大纲提取等）
- 只需添加新的消息处理逻辑

### 7. 错误处理和恢复

**决策**: 实现任务恢复机制

**策略**:
- 每个阶段完成后更新数据库状态
- 支持从失败点重试（如存储阶段失败后，可重新执行存储而不重做解析）
- 保留临时文件直到任务完成或取消
- 提供详细的错误日志

**理由**:
- 大文件处理耗时长，完全重试成本高
- LLM 调用可能因网络问题失败，需要重试机制
- 提升用户体验和系统可靠性

## Risks / Trade-offs

**[Risk] LLM API 成本** → **Mitigation**:
- AI 章节识别仅在用户明确选择时启用
- 提供"快速导入"（仅正则解析）和"智能导入"（AI 增强）两种模式
- 实现 LLM 调用缓存，相同文本内容复用结果
- 支持配置 Provider 优先级，优先使用便宜或免费的 Provider

**[Risk] 大文件处理超时** → **Mitigation**:
- 设置合理的超时时间（上传 30 分钟，处理 2 小时）
- 实现心跳检测，防止长时间无响应被代理断开
- 支持任务恢复，超时后可从断点继续

**[Risk] 内存不足** → **Mitigation**:
- 使用流式读取，不将整个文件加载到内存
- 分批次处理章节，批量插入数据库
- 配置最大文件大小限制（如 500MB）
- 监控内存使用，超过阈值时拒绝新任务

**[Risk] 并发任务过多** → **Mitigation**:
- UnifiedAgentScheduler 已实现资源限制
- 限制同时运行的导入任务数量（如 2 个）
- 新任务进入队列等待，前端展示排队位置

**[Trade-off] 实时性 vs 系统复杂度**:
- WebSocket 实时进度增加系统复杂度
- 但大幅提升用户体验
- 决策：接受复杂度，提供实时进度

**[Trade-off] AI 准确性 vs 处理速度**:
- AI 章节识别更准确但慢（可能需要数分钟）
- 正则解析快速但不够智能
- 决策：提供两种模式让用户选择

## Migration Plan

### 部署步骤

1. **Phase 1: 后端模块迁移**
   - 将 AI 文本导入工具核心逻辑迁移到 `sail_server/utils/text_import/`
   - 确保现有脚本仍可独立运行（向后兼容）
   - 添加单元测试

2. **Phase 2: 异步导入 API**
   - 实现 `/api/v1/text/import-async` 端点
   - 扩展 UnifiedAgentScheduler 支持导入任务类型
   - 集成 WebSocket 进度通知

3. **Phase 3: 前端改造**
   - 实现流式文件上传组件
   - 添加导入任务面板
   - 集成 WebSocket 消息处理

4. **Phase 4: 集成测试**
   - 测试各种大小文件（1MB, 10MB, 100MB, 1GB）
   - 测试不同编码格式（UTF-8, GBK, GB2312）
   - 测试边界情况（空文件、无章节文本、超大章节）

### 回滚策略

- 数据库无需迁移，可立即回滚
- 保留现有同步导入 API，新旧方式并存
- 前端可通过特性开关切换新旧导入方式
- 临时文件定期清理，不污染系统

## Open Questions

1. **文件大小限制**: 是否需要配置化最大文件大小？当前建议 500MB。

2. **并发限制**: 同时运行导入任务的最大数量建议为 2 个，是否合理？

3. **AI 模式默认开启**: 智能导入是否应该作为默认选项？考虑到成本和速度，建议默认关闭。

4. **历史任务保留**: 导入任务历史记录保留多长时间？建议 30 天后自动清理。

5. **章节预览**: 导入完成后是否需要提供章节结构预览，让用户确认后再正式保存？

6. **编码检测**: 是否使用 `chardet` 库自动检测编码？建议实现，但允许用户覆盖。
