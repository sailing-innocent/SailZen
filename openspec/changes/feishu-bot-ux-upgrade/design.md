## Context

当前 `bot/feishu_agent.py` 实现了基本的 Feishu-OpenCode 桥接功能，但交互体验较为原始。用户反馈的三个核心问题：

1. **会话生命周期管理混乱**: 机器人启动/关闭时无法正确管理 OpenCode 进程状态，导致僵尸进程或状态不一致
2. **纯文本交互局限**: 当前仅支持文本回复，无法展示结构化信息（如工作区列表、会话状态、进度指示）
3. **缺少反馈和确认**: 长时间操作无等待提示，高风险操作无二次确认，用户体验生硬

本设计基于之前 `redesign-feishu-opencode-bridge-workflow` 的基础架构，专注于 UI/UX 层面的全面升级。

## Goals / Non-Goals

**Goals:**
- 实现机器人与 OpenCode 会话的完整生命周期绑定（启动、恢复、关闭、异常处理）
- 使用飞书消息卡片替代 80% 以上的纯文本交互
- 为所有耗时操作（>1s）添加进度指示或等待状态
- 为高风险操作添加显式的二次确认流程
- 保持命令语义的向后兼容性

**Non-Goals:**
- 不重构底层的 OpenCode 进程管理逻辑（复用现有实现）
- 不引入新的 AI 意图识别（复用现有的 BotBrain）
- 不实现实时双向同步（保持现有的轮询/响应模型）
- 不构建独立的移动端应用

## Decisions

### 1. 卡片优先的交互模型

**Decision**: 使用飞书消息卡片作为主要交互媒介，仅在初始化或错误时 fallback 到文本。

**Rationale**:
- 卡片支持按钮、颜色、排版等丰富的视觉元素
- 支持动态更新（更新同一条消息而非发送新消息）
- 移动端体验远优于纯文本

**Alternatives considered**:
- 纯文本 + 富文本标记：飞书对 Markdown 支持有限，交互性差
- 独立 Web 控制台：超出了本次迭代的范围

### 2. 会话状态机的显式建模

**Decision**: 为每个工作区维护显式的状态机：`idle` → `starting` → `running` → `stopping` → `idle`/`error`

**Rationale**:
- 状态转换的可视化让用户了解当前进展
- 避免竞态条件（如重复启动、关闭中接收命令）
- 支持从异常状态恢复

**State Transitions**:
```
idle --[启动命令]--> starting --[健康检查通过]--> running
running --[停止命令]--> stopping --[进程退出]--> idle
any --[异常/超时]--> error --[恢复命令]--> starting
```

### 3. 渐进式确认策略

**Decision**: 根据操作风险等级采用不同的确认策略

**Rationale**:
- 低风险操作（查看状态）直接执行，保持流畅
- 中风险操作（启动新会话）仅在资源紧张时提示
- 高风险操作（停止会话、切换工作区）强制确认

**Risk Levels**:
- `safe`: 查看、查询类操作 - 无需确认
- `guarded`: 启动、创建类操作 - 条件确认
- `confirm-required`: 停止、删除、修改类操作 - 强制确认

### 4. 状态持久化与恢复

**Decision**: 将活跃会话状态持久化到文件，支持机器人重启后自动恢复

**Rationale**:
- 避免重启后丢失用户上下文
- 支持优雅关闭时保存状态
- 启动时自动重连活跃会话

**Storage**: `~/.config/feishu-agent/active_sessions.json`

### 5. 异步操作的可视化反馈

**Decision**: 耗时操作先发送"处理中"卡片，完成后更新为结果卡片

**Rationale**:
- 用户知道系统正在工作而非卡住
- 飞书支持消息更新，避免刷屏
- 可展示进度百分比（如果底层支持）

**Pattern**:
1. 接收命令 → 立即发送"处理中"卡片
2. 执行操作 → 更新卡片显示进度
3. 操作完成 → 更新卡片为最终结果

## Risks / Trade-offs

- [卡片依赖飞书 API 可用性] → Mitigation: 保留文本 fallback，API 失败时降级
- [频繁卡片更新可能触发频率限制] → Mitigation: 批量更新，最少 3 秒间隔
- [状态机复杂性增加] → Mitigation: 使用简单的状态枚举，单元测试覆盖转换逻辑
- [会话恢复失败可能静默失败] → Mitigation: 恢复失败时发送通知卡片，让用户手动处理

## Migration Plan

**部署步骤**:
1. 代码更新: 更新 `bot/feishu_agent.py`
2. 配置更新: 确认飞书应用有消息卡片权限
3. 数据准备: 创建 `~/.config/feishu-agent/` 目录
4. 滚动重启: 新机器人启动时会话状态自动恢复

**Rollback**:
- 回退到上一版本代码
- 旧版本兼容现有会话状态文件格式
